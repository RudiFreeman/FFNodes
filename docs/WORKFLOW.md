Регламент, по которому ведётся **любая** работа над проектом. Облегчённая версия:
суть процесса без тяжёлой спринт-бюрократии — под раннюю стадию проекта.
Для кого: Claude Code + Алексей. Когда читать: **перед каждой задачей, до первой правки.**

> 🔒 Claude сверяется с этим документом в начале задачи и **напоминает** о порядке.
> Отклонение допускается **только** при явном указании Алексея «идём в обход» по
> конкретному шагу — и Claude проговаривает это вслух («отступаем от WORKFLOW: пропускаем
> шаг X по твоему решению»), чтобы оно было осознанным.
>
> Принцип: **лучше дольше, но чище** — даже мелкая правка идёт по порядку.

## 0. Словарь терминов (чтобы говорить на одном языке)

Три уровня работы, от большого к малому. Алексею **не нужно** помнить git-термины —
достаточно говорить естественно («закрыли спринт», «дай следующий спринт», «сделай первый
пункт»), Claude переводит в git сам. Эта таблица — общая шпаргалка.

| Уровень | Что это | Слово | Git |
|---|---|---|---|
| **Большой** | Весь приоритезированный план хотелок целиком | **роадмап** | интеграционная ветка `feature/<роадмап>` (напр. `feature/ux-roadmap`) |
| **Средний** | Один крупный кусок работы — несколько связанных задач | **спринт** | ветка спринта `feat/sprint-N-...` |
| **Малый** | Одна конкретная задача внутри спринта | **пункт** | один коммит |

```
роадмап  (feature/ux-roadmap)
 ├─ Спринт 1  (feat/sprint-1-ux)
 │   ├─ Пункт 1  → коммит
 │   ├─ Пункт 2  → коммит
 │   └─ Пункт 3  → коммит
 └─ Спринт 2  (feat/sprint-2-...)
     ├─ Пункт 1  → коммит
     └─ Пункт 2  → коммит
```

> ⚠️ **Подветок нет.** Один спринт = **одна ветка**; пункты внутри — это **коммиты**,
> а не отдельные ветки. Структура плоская в две ветки (интеграционная + ветка спринта).
> Спринты нумеруются числами (Спринт 1, 2, 3…), см. §5.

## 1. Когда применяется

На **любое** изменение, попадающее в репозиторий: код, конфиги, ассеты, документация.

## 2. Порядок шагов

```
план → 🚦 утверждение → ветка → dev → 🚦 утверждение → test → review → fix → доки → коммит
```

### Шаг 1 — Планирование (до кода)
- Claude изучает контекст (CLAUDE.md, релевантные доки, код), формулирует задачу,
  находки, риски, задаёт уточнения.
- 🚦 Алексей **утверждает план**. Без утверждения кода не пишем.

### Шаг 2 — Ветка
- Работаем **только в отдельной ветке**, не в `main`.
- Имя: `feat/<кратко>` для новой фичи, `fix/<кратко>` для багфикса.
- ⚠️ Ветку создаём **до первой строки кода**. Проверь `git branch --show-current` перед dev.

> 🌿 **Модель веток для роадмапа спринтов (Модель Б — сборная ветка).**
> Группа связанных спринтов копится в интеграционной ветке `feature/<роадмап>`
> (напр. `feature/ux-roadmap`), а в `main` уходит **одним вливанием в конце**, когда
> все спринты группы готовы.
> - Каждый спринт ветвится **от интеграционной ветки**, не от `main`:
>   `git checkout feature/ux-roadmap && git checkout -b feat/sprint-N-...`.
> - Закрытый спринт вливается обратно в интеграционную ветку (`--no-ff`).
> - ⚠️ Чтобы не копить поздние конфликты: держим интеграционную ветку свежей и
>   ветвим следующий спринт **от неё** (Спринт N видит код Спринта N−1).
> - Финал: `feature/<роадмап>` → PR → `main` (мерж за Алексеем).

### Шаг 3 — Цикл реализации

```
dev → 🚦 утверждение → test → review → fix
```

| Фаза | Что делаем | Скил / инструмент |
|---|---|---|
| **dev** | пишем код; фронт проверяем в браузере/приложении | `feature-dev` (+ `frontend-design` для UI), `run` / `verify` |
| 🚦 **утверждение** | показываем результат → Алексей подтверждает | — |
| **test** | пишем тесты, обновляем реестр `tests/README.md` | Vitest (фронт) / тесты Rust при необходимости |
| **review** | независимое код-ревью | `/code-review` |
| **🔒 security-review** | если задача трогает security-поверхность (см. ниже) | `/security-review` |
| **fix** | чиним находки + причёсываем; тесты до зелёного | `/simplify` |

> 🆕 **`/feature-dev` — для нового, если уместен.** Когда dev-фаза создаёт что-то новое
> (экран, подсистему, заметную фичу), скил сверяет это с паттернами кодовой базы. Для мелких
> точечных правок не нужен.

> 🔒 **`/security-review` — обязателен**, если задача затрагивает: запуск внешних процессов
> (вызов FFmpeg с пользовательским вводом), приём произвольных путей/файлов, доступ к файловой
> системе через Tauri, любые секреты/ключи. Итог фиксируется (находки закрыты либо «дыр нет»).

### Шаг 4 — 🟢 Актуализация документации (часть Definition of Done)
**Нельзя пропускать — именно из-за этого протухают доки.** Перед коммитом Claude проходит
карту §4 и обновляет всё, что задача затронула.

### Шаг 5 — Коммит / PR
- Коммит по соглашению (§5), PR против `main`.
- Мерж в `main` делает Алексей. После мержа `main` локально не обновляется сам:
  `git checkout main && git pull origin main`, затем можно удалить влитую ветку.

## 3. Definition of Done (чек-лист перед «готово»)

- [ ] dev-часть утверждена Алексеем;
- [ ] тесты написаны и зелёные; `tests/README.md` обновлён;
- [ ] код-ревью пройдено, HIGH/MEDIUM закрыты;
- [ ] 🔒 если тронута security-поверхность (запуск FFmpeg/ввод→процесс·файлы, доступ к ФС,
      секреты) — `/security-review` проведён, находки закрыты либо «дыр нет»;
- [ ] для фронта: lint + **`npm run build` зелёный** (а не только `tsc --noEmit`);
- [ ] документация актуализирована по карте §4;
- [ ] изменения в отдельной ветке, готов PR.

## 4. Карта актуализации документации

| Что изменилось | Обязательно обновить |
|---|---|
| Завершён этап / выпущена версия | [../CHANGELOG.md](../CHANGELOG.md) + раздел «Статус и этапы» в [../CLAUDE.md](../CLAUDE.md) |
| Новая/изменённая структура папок, модулей | дерево и разделы [../CLAUDE.md](../CLAUDE.md) |
| Новая папка-хранилище | `README.md` внутри неё + строка в карту [README.md](README.md) |
| Изменилось продуктовое видение / скоуп | [PRD.md](PRD.md) |
| Написаны/нужны тесты | `../tests/README.md` |
| Замечен техдолг/странность не по теме задачи | `CODE_NOTES.md` — **не чинить на месте** |
| 🔒 Тронута security-поверхность | провести `/security-review`, итог зафиксировать |

> Версию (`vX.Y.Z`) бампаем только при завершении заметного этапа, не в процессе.

## 5. Соглашения

- **Claude напоминает и требует.** В начале задачи — сверка с этим документом. Если Алексей
  просит «давай быстро в обход» — Claude называет, какие шаги пропускаются, и продолжает
  только после явного «да, в обход».
- **Conventional Commits:** `feat` / `fix` / `test` / `docs` / `refactor` / `chore`,
  область в скобках — `feat(nodes):`, `test(ffmpeg):`, `docs(workflow):`.
- **Идентификаторы — латиница и цифры** (этапы `s1`, `s2`…). Не использовать кириллицу
  в идентификаторах (визуально путается: `B`↔`В`, `A`↔`А`).
- **Спринты нумеруются числами** — Спринт 1, Спринт 2, Спринт 3, … (а не буквами A/B/C).
  Так договорились с Алексеем: число читается однозначно и задаёт явный порядок.
- **Ничего не пушим в `main` напрямую**; мерж — за Алексеем.
