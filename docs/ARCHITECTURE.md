# Архитектура — FFmpeg Visual

> Как граф нод превращается в корректную FFmpeg-команду и исполняется.
> Стек: Tauri (Rust-оболочка) + React/TypeScript (UI + логика) + React Flow (холст).
> Статус: черновик v0.2. Реализован **полноценный DAG** — несколько входов, ветвление и
> слияние потоков (`-filter_complex`). Линейная цепочка (один вход → фильтры → выход) —
> частный случай, идёт прежним `-vf` путём. Рубильник — `dag.isLinearGraph`.

---

## 1. Принцип: однонаправленный поток данных

```
   Граф (ноды + связи)
          │  пользователь редактирует на холсте
          ▼
   GraphModel  ──────► validate() ──────► generateCommand() ──────► CommandString
   (состояние)          проверка            чистая функция           (массив аргументов)
          │                                                              │
          │                                                  показываем внизу (редактируема)
          ▼                                                              ▼
   React Flow UI                                              Tauri.invoke('run_ffmpeg', args)
                                                                         │
                                                                         ▼
                                                              Rust запускает FFmpeg-процесс
                                                                         │
                                                          stdout/stderr/progress ──► UI (превью, лог)
```

Ключевое: **граф → команда** — это поток в одну сторону через чистые функции (легко тестировать).
Обратный разбор «команда → граф» (когда профи правит строку руками) — отдельная большая задача,
вынесена за MVP (см. [IDEAS.md](IDEAS.md), раздел «Двусторонняя синхронизация»).

## 2. Модель данных графа

Минимальный, но расширяемый набор типов. Живёт в `src/shared/types/graph.ts`.

```ts
// Тип ноды определяет её роль во FFmpeg-команде
type NodeKind = 'input' | 'filter' | 'output';

interface GraphNode {
  id: string;                 // уникальный (React Flow требует)
  kind: NodeKind;
  filterId?: string;          // для filter: ссылка на запись каталога ('scale', 'fps'...)
  params: Record<string, ParamValue>;  // значения параметров (заполнены пользователем)
  position: { x: number; y: number };  // координаты на холсте
}

interface GraphEdge {
  id: string;
  source: string;             // id ноды-источника
  target: string;             // id ноды-приёмника
  // на будущее (multi-input): какой выход/вход соединяем
  sourceHandle?: string;
  targetHandle?: string;
}

interface Graph {
  nodes: GraphNode[];
  edges: GraphEdge[];
}
```

> **Почему так под multi-input:** `sourceHandle`/`targetHandle` уже в модели — в MVP не используются
> (одна связь), но когда добавим overlay/concat, именно они станут метками `[0:v]`/`[tmp]` в
> `-filter_complex`. Менять структуру не придётся — только генератор.

## 3. Каталог фильтров (источник правды по функциям)

Каталог — **данные, а не код**. Каждая запись описывает один FFmpeg-фильтр/операцию:
как он называется по-человечески, что делает, какие параметры, как кладётся в команду.

```ts
interface FilterParam {
  id: string;                 // 'width', 'fps'...
  label: string;              // «Ширина» — человеческое имя
  type: 'number' | 'string' | 'enum' | 'boolean';
  default?: ParamValue;
  options?: string[];         // для enum
  hint?: string;              // helper-текст под полем
}

interface FilterDef {
  id: string;                 // 'scale'
  category: string;           // 'Размер / FPS'
  label: string;              // «Изменить размер»
  description: string;        // «Меняет разрешение видео. Зачем: ...» — ключевая ценность
  params: FilterParam[];
  // как собрать кусок команды из значений параметров:
  toFilterString: (p: Record<string, ParamValue>) => string;  // 'scale=1280:-2'
}
```

- Живёт в `src/shared/lib/ffmpeg/catalog/` — по файлу на категорию (convert, resize, trim, gif, audio…).
- Каталог справа (UI) рендерится **из этих данных** — добавил фильтр в каталог = появился в UI.
- Описания «что и зачем» пишем вручную (качество важнее полноты) — см. открытый вопрос в IDEAS.

## 4. Генерация команды (сердце проекта)

Модуль `src/shared/lib/ffmpeg/generate.ts` — **чистые функции, без побочных эффектов**.

### Шаги

1. **Топологическая сортировка** графа: input → filters по порядку связей → output.
   Для MVP (линейная цепочка) это просто проход по единственному пути.
2. **Сборка аргументов** в массив (НЕ склеенная строка — защита от инъекций, см. §6):
   - входы: `['-i', inputPath]`
   - видеофильтры: собрать `-vf "scale=1280:-2,fps=15"` (запятая склеивает цепочку)
   - аудиофильтры: собрать `-af "volume=2"` (отдельная цепочка, после `-vf`)
   - выходные опции (кодек, формат) из output-ноды
   - выходной путь
3. **Результат:** `{ args: string[], display: string }` — `args` для запуска, `display` для показа внизу.

### Два пути генерации (реализовано: этап multi-input)

| | Линейный граф (`-vf`) | DAG (`-filter_complex`) |
|---|---|---|
| Когда | один вход, нет merge-операций, нет ветвления | merge-операция / >1 входа / ветвление-слияние |
| Опция фильтров | `-vf "f1,f2,f3"` | `-filter_complex "[0:v]f1[v1];[v1][1:v]overlay[vout]"` |
| Метки пэдов | не нужны | аллокатор лейблов (`complex/labels.ts`): `v1`/`a1`/`0:v` |
| Сортировка | линейный проход (`chain.orderedFilters`) | топосорт DAG (`dag.topoSort`, алгоритм Кана) |
| Входы | один `-i` | несколько `-i` + `-map` выходного потока |

**Рубильник — `dag.isLinearGraph(graph)`**: единственная точка решения, по какому пути идти.
Линейный путь не изменился (27 single-фильтров и их тесты не тронуты). DAG-путь строит
`complex/build.ts` (`buildComplexPlan`): топо-обход, обычный фильтр оборачивается в лейблы
`[prev]vf[next]`, merge-операция строит фрагмент сама через `MergeSpec.toComplex`. Порядок
входов merge-ноды разрешается по `GraphEdge.targetHandle`. Генератор изолирован — `generate.ts`
делегирует DAG-случай в `generateComplexCommand`.

### Предсказание характеристик (вторая чистая функция)

Рядом с генератором — `src/shared/lib/ffmpeg/predict.ts`: `predictOutput(graph, info)`.
Если генератор даёт «граф → аргументы команды», то предсказание — «граф + вход → итоговые
характеристики» (разрешение, fps, длительность, кодек, формат). Питает превью «До → После»
на нодах и в боковой панели, пересчитывается на лету (до рендера).

- Обход цепочки (`orderedFilters`) вынесен в общий `src/shared/lib/ffmpeg/chain.ts` —
  единый источник правды о порядке для `generate.ts` и `predict.ts`.
- Каждая операция каталога описывает свой вклад полем `FilterDef.applyToInfo?(info, params)`
  (чистое преобразование `MediaInfo`). Операции без него характеристики не меняют.
- `size_bytes` точно не предсказывается (битрейт/CRF/контент) — в UI помечается «≈ оценка»,
  после рендера заменяется реальным ffprobe-значением (см. CODE_NOTES N-010).

## 5. Валидация (до запуска)

`src/shared/lib/ffmpeg/validate.ts` — проверяет граф **перед** генерацией, даёт понятные ошибки:

- есть ровно один input и один output (для MVP);
- граф связный, нет «висячих» нод;
- нет циклов;
- обязательные параметры фильтров заполнены;
- путь входного файла существует (проверка через Tauri).

Ошибки показываем рядом с проблемной нодой + в общем сообщении (правило `error-clarity`: причина + как починить).

## 6. Исполнение через Tauri (Rust-сторона)

`src-tauri/` предоставляет команды, вызываемые из фронта через `invoke()`.

| Tauri-команда | Что делает |
|---|---|
| `probe_media(path)` | вызывает `ffprobe` → метаданные (длительность, разрешение, кодеки) для превью |
| `run_ffmpeg(args)` | запускает FFmpeg как процесс, стримит прогресс/лог обратно в UI; кладёт процесс в общий `RenderState` (для отмены) |
| `cancel_render()` | убивает текущий FFmpeg-процесс (`kill`+`wait`), помечает отмену в `RenderState`; `run_ffmpeg` затем удаляет недописанный выходной файл |
| `extract_frame(input_path, vf, at_sec)` | FFmpeg `-frames:v 1` → JPG в temp-папке для превью-кадра; `vf` (из генератора) даёт кадр «После», без него — «До». Возвращает путь; webview читает через asset-протокол |
| `pick_input_file()` | системный диалог выбора файла |
| `find_ffmpeg()` | ищет бинарник FFmpeg (системный или бандленный) |

**🔒 Безопасность (см. WORKFLOW — security-review обязателен):**
- Аргументы передаём **массивом**, FFmpeg запускаем напрямую (не через `sh -c`) — нет shell-инъекций
  даже если в имени файла или параметре спецсимволы.
- Пути валидируем: путь с ведущим `-` префиксуется `./` (`safe_path`/`safePath`), чтобы
  ffmpeg/ffprobe не приняли его за флаг (argument injection, N-004); не даём команде уйти в shell.
- Tauri allowlist: разрешаем только нужные возможности (запуск конкретного бинарника, выбранные пути).
- Asset-протокол (превью-кадр) включён с **узким scope** `**/ffmpeg-visual-frame-*.jpg` —
  webview читает только наши кадры в temp, не произвольные файлы на диске.

## 7. Поток прогресса и превью

- FFmpeg пишет прогресс в stderr (`frame=`, `time=`). Rust парсит → шлёт события в UI (прогресс-бар на `Render`).
- По завершении — путь к результату → `PreviewPanel` показывает «после» (переключатель до/после).
- `probe_media` на входе → сразу показываем «до» и метаданные.
- **Превью-кадр:** `extract_frame` рендерит JPG-кадр (FFmpeg). «До» — из исходника при выборе
  файла; «После» — тот же момент через vf-цепочку графа (`videoFilterChain`), пересчёт с
  дебаунсом в хуке `usePreviewFrame`. Видеоплеер с проигрыванием — отдельный будущий этап.

## 9. Тестируемость

Сердце — `generate.ts` и `catalog/` — чистые функции и данные. Тесты (Vitest):
- граф → ожидаемый массив аргументов (table-driven: вход-граф → ожидаемая команда);
- каждый `FilterDef.toFilterString` → корректная строка фильтра;
- валидатор ловит циклы/висячие ноды/незаполненные параметры.

Rust-часть (`run_ffmpeg`) тестируем отдельно/вручную — она тонкая (запуск процесса + парс прогресса).

## 10. Ключевые архитектурные решения (резюме)

1. **Поток граф → команда однонаправленный**, через чистые функции. Обратный разбор — за MVP.
2. **Каталог фильтров — данные, не код.** UI и генерация читают один источник.
3. **DAG: линейный граф → `-vf`, сложный → `-filter_complex`** (рубильник `dag.isLinearGraph`).
   Merge-операции объявляют `FilterDef.merge` (MergeSpec); 27 single-фильтров не затронуты.
4. **Команда = массив аргументов**, не строка → нет shell-инъекций; FFmpeg через Tauri напрямую.
5. **Изоляция генератора** — переход на сложные графы не трогает UI и исполнение.
