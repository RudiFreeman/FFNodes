# Спринт 3 — Мульти-аутпут (1 вход → N выходов) — ПЛАН

Статус: на утверждении. Ветка: `feat/sprint-3-multi-output` (от `feature/ux-roadmap`).
Развилка утверждена: **Вариант A** (одна FFmpeg-команда, N `-map`/выходов).

---

## 0. Развилка A vs B — обоснование (утверждено: A)

**A — одна команда, N выходных секций.** Один декод входа → `split` потока → каждая ветка в
свою выходную секцию `-map [vN] -map 0:a <outputArgs> outN.mp4`.

Почему A ложится на наш код естественно:
- У нас уже есть `filter_complex`-путь (`complex/build.ts`) и приём с `split`-лейблами —
  ровно как GIF-палитра (один вход → split→palettegen→paletteuse). Мульти-аутпут — это
  тот же split, но ветки идут не в merge, а в РАЗНЫЕ выходы.
- `run_ffmpeg` (Rust) уже принимает готовый `args[]` и стримит один прогресс — формат args
  расширяется, контракт процесса не ломается.

Минусы A и как закрываем:
- Генератор усложняется: на каждый выход — свой блок `outputArgs` + свой путь. Решаем
  отдельной веткой генерации `generateMultiOutputCommand` (по аналогии с уже существующей
  `generateComplexCommand`), не трогая 27 single-фильтров линейного пути.
- Прогресс один на команду. Это НЕ минус для A: FFmpeg на одну команду с N выходами шлёт
  один общий `out_time_us` (по самому медленному выходу). Одна линия прогресса честно =
  «всё готово». N линий при A технически невозможны (один процесс не разделяет прогресс по
  выходам) — оставляем одну линию как сейчас.

Почему НЕ B: декод дублируется ×N (для 5 выходов 4K — заметно греет CPU и медленнее),
плюс Rust `RenderState` сейчас рассчитан на ОДИН процесс (`Mutex<Option<Child>>` + один
флаг отмены) — B потребовал бы переделку на список процессов и сведение прогресса/отмены.
A эффективнее и меньше трогает Rust-ядро.

## 1. Что в ядре завязано на «единственный выход» (карта правок)

- `dag.ts::topoSort` — `graph.nodes.find(n => n.kind === "output")` (ОДИН). → станет
  «все output-ноды», достижимость проверяется для каждого.
- `dag.ts::isLinearGraph` — неявно про один выход; добавим: ветвление в >1 выход → не линеен
  (идём complex/multi-output путём).
- `generate.ts` — оба пути строят один `PLACEHOLDER_OUTPUT` в конце. → новая ветка для N
  выходов.
- `complex/build.ts::buildComplexPlan` — мапит предшественника ЕДИНСТВЕННОГО output. →
  обобщим: план содержит СПИСОК выходов, каждый со своими map/outputArgs/путём.
- `predict.ts::predictOutput` — `ordered.find(n => n.kind === "output")` (ОДИН). → предсказание
  на ВЫБРАННЫЙ выход (или по каждому).
- `useGraph.ts` — фикс. `OUTPUT_ID="output"`, `deletable:false`, единственное ребро
  `input-output`, `addFilterNode` цепляет всё в один output. → поддержка N output-нод.
- `useRender.ts` — один `pickOutputFile`, один путь, один `outputInfo`. → N save-диалогов
  (по числу выходов), пути подставляются в плейсхолдеры по порядку выходов.
- Rust `run_ffmpeg` — `out_path = args.last()` для чистки при отмене. → при N выходах
  чистим ВСЕ выходные файлы (последние N аргументов / переданный список).

## 2. Модель графа (пункт 1)

`graph.ts`: ничего ломать не нужно — `NodeKind` уже включает `"output"`, граф допускает
несколько нод любого kind. Меняется лишь ДОПУЩЕНИЕ «output один».

Новое:
- Каждая output-нода хранит свои настройки кодирования в `params` (или ссылку на пресет
  выходного формата). На старте Спринта 3 — минимально: имя/формат + те outputArgs, что
  даёт цепочка перед ней. Полноценные «настройки на выход» (кодек/CRF/разрешение прямо в
  output-ноде) — отдельный пункт, см. §7 (решим объём ниже).
- `useGraph`: стартовая пара input→output сохраняется. Кнопка «Добавить выход» (в каталоге,
  рядом с «Добавить вход») создаёт новую output-ноду (`deletable:true`, основной output —
  `deletable:false`). `addFilterNode` для линейного случая по-прежнему цепляет в основной
  output; ветки на доп. выходы пользователь тянет сам (как доп. входы сейчас).
- `onConnect` уже корректно обрабатывает один источник на target-handle — для выходов это и
  нужно (в каждый выход ведёт одна ветка).

UI-холст: `OutputNode` получает `deletable` и, для доп. выходов, кнопку выбора имени файла /
индикатор «выход N». Регистрация типа уже есть.

## 3. Генератор (пункт 2) — Вариант A

- `dag.ts`: `topoSort` обобщаем на N выходов (собираем все `kind==="output"`, достижимость
  для каждого входа до ХОТЯ БЫ одного выхода + каждый выход достижим из входа). `isLinearGraph`
  возвращает false при >1 output.
- `complex/build.ts`: план становится мульти-выходным. Вместо одного `mapVideo/mapAudio`
  возвращаем `outputs: { mapVideo, mapAudio, outputArgs, nodeId }[]`. Для каждого выхода
  поднимаемся к его предшественнику и берём его потоки. Если две ветки идут от одного
  источника — аллокатор уже даёт разные лейблы; при необходимости вставляем `split`
  (когда один поток-источник кормит >1 потребителя) — это новый кусок логики аллокатора.
- `generate.ts`: новая `generateMultiOutputCommand(graph, inputPaths)`. Args:
  `-i ... -filter_complex "..." (-map v0 -map 0:a <out0Args> OUT0) (-map v1 ... <out1Args> OUT1)`.
  Плейсхолдеры выходов: `output_0`, `output_1`, … (фронт заменит на реальные пути по порядку).
  `display` — читаемый вид с короткими именами.
- `GeneratedCommand` расширяем: `outputCount` (или `outputPlaceholders: string[]`), чтобы
  `useRender` знал, сколько save-диалогов спросить и какие плейсхолдеры заменить.
- Линейный путь (один выход, один вход, без merge) — НЕ трогаем (27 фильтров работают как
  раньше). Мульти-аутпут активируется только при >1 output-ноде.

## 4. Прогресс + рендер (пункт 3) — Вариант A

- Одна линия прогресса остаётся (FFmpeg даёт один общий `out_time_us`). ProgressBar без
  изменений по смыслу.
- `useRender`: вместо одного `pickOutputFile` — по одному диалогу на каждый выход (имя по
  умолчанию `output_1.mp4`, `output_2.mp4`…). Если пользователь отменил любой диалог —
  отмена всего рендера. Подставляем пути в плейсхолдеры по порядку. `args` строятся как
  сейчас (замена последних N плейсхолдеров), `runFfmpeg` вызывается ОДИН раз.
- `outputInfo`: после рендера зондируем все выходные файлы → `outputInfos: MediaInfo[]`.
  Панель/нода показывает по выбранному выходу (см. пункт 4 превью).
- Rust `run_ffmpeg`: вместо `out_path = args.last()` принимаем явный `output_paths: Vec<String>`
  (или вычисляем все выходы) — при отмене чистим ВСЕ недописанные. 🔒 каждый путь уже
  прошёл `safePath` на фронте; в Rust — без изменений в safe_path, но чистим список.
  Контракт события render-progress/render-done не меняется.

## 5. Превью / predict (пункт 4)

- `predict.ts`: `predictOutput` принимает опц. `outputNodeId` — предсказание на конкретный
  выход; по умолчанию основной. Для панели «До/После» нужен выбор активного выхода.
- UI: если выходов >1 — над панелью «После» переключатель выходов (чипы «Выход 1 / 2 / …»);
  превью-кадр «После» и predict считаются для выбранного выхода. Реализация кадра «После»
  для конкретного выхода = тот же `extract_frame_complex`, но `map_video` = лейбл ветки
  выбранного выхода (построитель уже даёт лейблы по выходам).
- На старте можно сделать минимально: панель показывает ОСНОВНОЙ выход, переключатель —
  если успеваем в рамках пункта 4 (объём уточним по ходу, премиса не должна раздуть спринт).

## 6. Нарезка на пункты (коммиты)

1. **feat(graph): модель N выходных нод + холст/onConnect** — N output-нод, «Добавить выход»,
   deletable, OutputNode с выбором имени; `topoSort`/`isLinearGraph` на N выходов. Тесты dag.
2. **feat(ffmpeg): генератор мульти-аутпута (Вариант A)** — `buildComplexPlan` → список
   выходов + split-аллокатор; `generateMultiOutputCommand`; `GeneratedCommand.outputPlaceholders`.
   Тесты generate/complex/build.
3. **feat(render): прогресс и запуск на N выходов** — `useRender` N save-диалогов + подстановка;
   Rust `run_ffmpeg` чистит все выходы при отмене; `outputInfos`. Тесты Rust.
4. **feat(preview): predict/превью на выбранный выход** — `predictOutput(outputNodeId)`,
   переключатель выходов в панели, кадр «После» по выходу. Тесты predict.

(Точную нарезку финализируем — возможно слияние 3↔4 или вынос UI-полировки.)

## 7. Открытые под-вопросы (на утверждение вместе с планом)

- **Глубина «настроек на выход».** Минимум: каждый выход = ветка фильтров + имя файла
  (настройки кодека задаются обычными нодами «Сменить кодек»/«Сжать» в ветке). Максимум:
  у output-ноды свои поля кодек/CRF/контейнер. Предлагаю МИНИМУМ в Спринте 3 (ветки нод
  уже покрывают сценарий «1080p H.264 / 720p H.265»), богатую панель выхода — следующим
  спринтом. ← нужно твоё «ок».
- **Save-диалоги: N штук подряд или одна папка + авто-имена.** Предлагаю N диалогов подряд
  (контроль над каждым путём, как сейчас). Папка-назначение — на будущее.
- **Переключатель выходов в превью** — делаем в пункте 4 или выносим? Предлагаю лёгкую
  версию в пункте 4.
