# Changelog — FFmpeg Visual

История завершённых этапов. Формат: обратная хронология (новое сверху).

## [Не выпущено]

### Превью-кадр «До → После» (этап «preview-frame»)
- **Реальный кадр вместо заглушки:** панель превью показывает настоящий кадр из видео
  (раньше была иконка-плейсхолдер). «До» — кадр исходника, «После» — тот же момент,
  прогнанный через vf-цепочку графа. Переключатель «До/После» под кадром (появляется,
  только когда есть видеофильтры).
- **Новая Rust-команда `extract_frame(input_path, vf, at_sec)`** — запускает FFmpeg
  (`-ss … -i … [-vf …] -frames:v 1 -q:v 3`), пишет JPG во временную папку ОС, возвращает
  путь. Запуск напрямую (Command, не shell), аргументы массивом. Чистая сборка аргументов
  вынесена в `build_frame_args` (тестируема).
- **Asset-протокол Tauri** включён с узким scope `**/ffmpeg-visual-frame-*.jpg` — webview
  читает только наши кадры, не произвольные файлы. Кадр отдаётся в `<img>` через
  `convertFileSrc`.
- **Чистый модуль `shared/lib/ffmpeg/frame.ts`** — `videoFilterChain(graph)` собирает только
  vf-фрагменты (без кодек-опций, для кадра не нужны), переиспользует `orderedFilters`.
- **Хук `usePreviewFrame`** — «До» при смене файла, «После» с дебаунсом (400мс) при правке
  графа; токены актуальности против гонок; «После» зеркалит «До», когда фильтров нет.
- **Кадр снимается из середины видео** (информативнее часто-чёрного нулевого кадра).
- **Видеоплеер с проигрыванием — отдельный этап позже** (нативный `<video>` в webview не
  играет HEVC/Main 10; кадр через FFmpeg работает на всех форматах).
- Техдолг: N-011 (кадры копятся в temp), N-012 (кадр «После» для временны́х фильтров
  trim/speed может быть нерепрезентативен) → CODE_NOTES.
- Тесты: фронт 52 ✅ (+5: `videoFilterChain`), Rust 9 ✅ (+3: `build_frame_args`).

### Превью характеристик «До → После» (этап «preview-before-after»)
- **Вторая чистая функция рядом с генератором:** `shared/lib/ffmpeg/predict.ts` —
  `predictOutput(graph, info)` прогоняет вход через цепочку фильтров и считает итоговые
  характеристики (разрешение, fps, длительность, кодек, формат). Пересчёт на лету при
  изменении графа. Обход цепочки `orderedFilters` вынесен из `generate.ts` в общий
  `shared/lib/ffmpeg/chain.ts` (один источник правды о порядке для генератора и предсказания).
- **Модель каталога:** `FilterDef.applyToInfo?` — необязательное чистое преобразование
  характеристик. Прописано операциям, которые их реально меняют (scale/crop/rotate —
  разрешение; fps; trim/speed — длительность; convert/gif/аудио — кодек/формат). Операции
  без него (цвет, отражение, 180°) характеристики не трогают.
- **UI на нодах:** ноды «Вход» и «Выход» показывают базовую сводку (разрешение, fps, кодек,
  размер) через общий `shared/ui/MediaSummary`. Вход — реальные данные файла, выход —
  предсказание; пересчитывается при добавлении/правке нод.
- **Боковая панель:** характеристики «До/После» сгруппированы (Видео / Звук / Файл) для
  читабельности. Колонка «После» = живое предсказание до рендера, реальные ffprobe-метаданные
  после успешного рендера (тогда размер точный, без «≈»).
- **Честность размера:** `size_bytes` точно не предсказывается (зависит от битрейта/CRF/контента) —
  до рендера помечается «≈ оценка». N-010 → CODE_NOTES.
- `MediaInfo` перенесён в `shared/types/media.ts` (доменный тип; `lib/ffmpeg` не зависит от
  слоя `api`), реэкспорт из `api/tauri.ts` для совместимости.
- **Полировка вида (по фидбэку):**
  - `probe_media` (Rust) расширен до полного полезного набора полей ffprobe (мусор вроде
    codec_tag/disposition/time_base отсеян): видео — полное имя кодека, профиль, битрейт
    (fallback на контейнер), соотношение сторон, формат пикселей, цветовое пространство,
    число кадров; звук — полное имя, битрейт, частота, каналы, раскладка, формат сэмплов;
    файл — полное имя контейнера, число потоков, энкодер и дата создания (из tags).
    `MediaInfo` (Rust + TS) расширен; форматтеры `formatBitrate`/`formatSampleRate`/
    `formatChannels`/`bitDepthFromPixFmt`/`formatDate`. Поля только-про-файл (энкодер, дата,
    потоки) в таблице без колонки «После».
  - Панель: таблица «До/После» на CSS-grid с выровненными колонками, линиями-разделителями между
    группами, шире (`w-80`); «После» подсвечивается только при отличии от «До».
  - Ноды «Вход»/«Выход»: к значениям добавлены короткие подписи (разрешение/кадры/кодек/размер).
  - Холст: меньший стартовый масштаб (`fitViewOptions.maxZoom 0.75`), ноды не разворачиваются
    на весь экран при открытии.
- Тесты: фронт 47 ✅ (было 28; +5 chain + 14 predict), Rust 6 ✅ (+1 на fallback битрейта).
  Typecheck и production-сборка зелёные.

### Расширение каталога: гибкая модель + категория «Конвертация / Сжатие» (этап «catalog-expand»)
- **Модель каталога расширена:** `FilterDef.toFilterString` → `toCommand`, возвращающий
  гибкий `CommandContribution { vf?, outputArgs? }`. Операция может вкладывать видеофильтр
  в `-vf` И/ИЛИ выходные опции-флаги (`-c:v`, `-crf`, `-an`, `-vn`). Покрывает не только
  видеофильтры. Существующие scale/fps/trim/crop переведены на новый контракт (→ `{ vf }`).
- **Генератор** собирает оба вида: vf-фрагменты в цепочку `-vf "f1,f2"` + outputArgs в правильном
  порядке (вход → -vf → выходные опции → выход). Тесты на комбинацию и только-outputArgs.
- **Новая категория «Конвертация / Сжатие»** (`catalog/convert.ts`): «Сжать видео» (CRF/H.264),
  «Извлечь аудио» (-vn -c:a copy), «Убрать звук» (-an) — с человеческими описаниями.
- Тесты: 23 ✅ (было 18; +5 на модель и операции). Проверено вживую: категория в каталоге,
  команда собирается с outputArgs (`-c:v libx264 -crf 23`) и в комбинации с `-vf`.
- N-007 (несочетаемые операции не валидируются) → CODE_NOTES.
- **Каталог — сворачиваемые группы-аккордеоны:** клик по заголовку категории сворачивает/
  разворачивает группу (шеврон + счётчик функций); несколько групп открыто одновременно,
  по умолчанию все развёрнуты; при поиске группы форсятся открытыми. Проверено вживую.
- **+4 категории (7→13 операций):** Поворот / Отражение (повернуть 90/180°, отразить),
  Скорость (ускорить/замедлить, setpts), Цвет / Яркость (eq + ч/б), Экспорт GIF
  (fps+scale+`-f gif`). Все с человеческими описаниями. GIF проверен на реальном файле
  (валидный GIF 89a). N-008 (GIF без палитры), N-009 (скорость не трогает звук) → CODE_NOTES.
- **Избранное + категории свёрнуты по умолчанию:** все аккордеоны теперь закрыты при старте
  (функций много). Секция «Избранное» вверху каталога — всегда открыта, отделена полосой;
  звёздочка у каждой функции (как эффекты в Premiere) копирует её в избранное. Хранится в
  localStorage (`useFavorites`), переживает перезапуск. Проверено вживую + персистентность.
  Избранное открыто по умолчанию, но сворачивается кликом по заголовку (шеврон).

### Рендер + линия прогресса (этап «render») — полный цикл «файл → результат»
- **Rust `run_ffmpeg`:** запускает FFmpeg (🔒 напрямую, args массивом) с `-progress pipe:1`,
  парсит `out_time_us` (чистая `parse_progress_seconds`, +1 тест), стримит процент во фронт
  событиями `render-progress`/`render-done`. Учитывает длительность входа для расчёта %.
- **Фронт:** `shared/api/tauri.ts` (+`pickOutputFile` save-диалог, `runFfmpeg`, `onRenderProgress`),
  `features/run-render/useRender` (статус idle/running/done/error, подписка на прогресс,
  подстановка выбранного выходного пути в args).
- **UI:** кнопка «Рендер» оживлена (onClick, спиннer, disabled без файла/при ошибке графа);
  `widgets/ProgressBar` — тонкая зелёная линия под TopBar, видна только во время рендера.
- 🔒 security-review: серьёзных дыр нет (прямой запуск, пути из системных диалогов).
  N-004 расширен на run_ffmpeg; N-005 (нет отмены рендера) добавлен в CODE_NOTES.
- ⚠️ Полный клик (Рендер → save-диалог → файл) проверяется вручную (нативный диалог);
  команда+прогресс доказаны прогоном на реальном Test_video.mov (640×1138, 30fps, 51→22 МБ,
  `out_time_us`+`progress=end` как ожидает парсер).
- **Доработка прогресса:** подписка на `render-progress` ставится ДО запуска (не теряем
  ранние события быстрого рендера); линия прогресса видна и при `done` (100%, не мелькает).
- **Фикс «полоса прыгала на 100% в конце»:** `run_ffmpeg` сделана `async` — синхронная
  Tauri-команда блокировала главный поток и копила события прогресса до возврата (приходили
  пачкой в конце). Теперь команда на отдельном потоке, события доставляются по ходу → полоса
  растёт в процессе рендера.
- **Метаданные результата (До/После):** после рендера output прозондируется (`probeMedia`),
  `PreviewPanel` показывает две колонки — характеристики входа и результата рядом
  (разрешение, длительность, FPS, размер). Сброс «После» при новом рендере.

### Выбор файла + метаданные (этап «file-probe», первый Rust-код)

- Подключён плагин `tauri-plugin-dialog` (+ разрешение `dialog:default` в capabilities).
- **Фронт:** `shared/api/tauri.ts` (`pickInputFile`, `probeMedia` + тип `MediaInfo`),
  `features/input-file/useInputFile` (выбор файла → probe → состояние), `shared/lib/format.ts`
  (длительность/байты/имя файла). `PreviewPanel` переписан: кнопка «Открыть файл» + панель
  характеристик под превью (идея из IDEAS реализована).
- Генератор принимает реальный путь: `args` — полный путь (запуск), `display` — короткое имя.
- 🔒 security-review проведён: серьёзных дыр нет (прямой запуск, минимальные разрешения,
  только чтение). N-004 (путь с `-` → профилактика) в CODE_NOTES. FFmpeg установлен (brew).
- ⚠️ Сквозной путь (диалог → метаданные в окне) проверяется вручную — нативный диалог Playwright

### Генератор команды + параметры нод (этап «command-generator»)
- **Сердце проекта:** `src/shared/lib/ffmpeg/generate.ts` — чистая функция `Graph → {args, display}`.
  Обходит граф от input по связям до output (MVP — линейная цепочка, опция `-vf`), собирает фильтры
  в порядке связей. Команда = массив аргументов (защита от инъекций). Неполный/циклический граф →
  понятная ошибка, без падения. Тесты — Vitest table-driven (8: цепочки, разрыв, цикл, без фильтров).
- Ноды `InputNode` (синяя) / `OutputNode` (зелёная); `FilterNode` получил редактируемые поля параметров.
- `useGraph`: стартовая пара вход→выход (неудаляемые); добавленный фильтр встраивается в цепочку
  перед выходом с авто-перецепкой связей; `onParamChange` пишет значения в `data.params`; маппинг
  состояния React Flow → доменный `Graph`; генерация команды через `useMemo`.
- `CommandBar` показывает реальную сгенерированную команду (или подсказку при неполном графе),
  копирование в буфер; обновляется в реальном времени.
- Проверено вживую (Playwright): клик→фильтр в цепочке, правка параметра → команда меняется
  мгновенно (`scale=640:-2,fps=30`), консоль чистая. N-003 (удаление среднего фильтра рвёт цепочку) — в CODE_NOTES.

### Каталог фильтров + добавление нод (этап «catalog-nodes»)
- Модель данных графа — `src/shared/types/graph.ts` (`GraphNode`/`GraphEdge`/`Graph`,
  с заделом под multi-input через `sourceHandle`/`targetHandle`).
- Каталог фильтров как ДАННЫЕ (`src/shared/lib/ffmpeg/catalog/`): типы `FilterDef`/`FilterParam`,
  4 фильтра с человеческими описаниями (изменить размер, сменить FPS, обрезать, кадрировать),
  у каждого — `toFilterString`. Индекс с `getFilterDef`/`catalogByCategory`.
- `FilterCatalog` рендерится из каталога-данных (категории, поиск по имени/описанию,
  описание «что и зачем» под пунктом); клик добавляет ноду.
- `useGraph` (feature `add-node`) — состояние нод/связей React Flow + `addFilterNode`;
  id через `crypto.randomUUID()`, позиция со сдвигом по числу нод.
- Кастомная нода `FilterNode` (фиолетовая, цвет по типу из UI.md) с хэндлами.
- Тесты каталога — Vitest (9 ✅), реестр в `tests/README.md`.
- Проверено вживую (Playwright): клик в каталоге → нода на холсте, ошибок в консоли нет.
- Фикс N-002: заголовок окна → «FFmpeg Visual». N-001 (стиль контролов React Flow) — в CODE_NOTES.
