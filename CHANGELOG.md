# Changelog — FFNodes

История завершённых этапов. Формат: обратная хронология (новое сверху).

## Инфраструктура

### CI-сборка бандлов под 3 ОС (этап «ci-release-builds»)
- **GitHub Actions собирает FFNodes под macOS / Windows / Linux и кладёт бинари в Release.**
  Раньше Release v1.0.0 был без бинарей (сборка под 3 ОС с одного Mac невозможна — нужен CI).
  Теперь `.github/workflows/release.yml`: push тега `v*` (наш релизный флоу) или ручной запуск
  (`workflow_dispatch`, поле `tag` — для проверки CI без боевого тега) собирает и публикует.
- **Матрица 4 job'а:** macos-latest (Apple Silicon arm64), macos-13 (Intel x86_64),
  ubuntu-22.04 (Linux), windows-latest. На Linux ставятся системные зависимости Tauri 2
  (`libwebkit2gtk-4.1-dev`, `libappindicator3-dev`, `librsvg2-dev`, `patchelf` и пр.) через
  `apt-get`. Сборка — `tauri-apps/tauri-action@v0` (сам зовёт `npm run build` + `tauri build`,
  создаёт/обновляет Release, грузит ассеты). `bundle.targets: "all"` даёт `.dmg`/`.AppImage`/
  `.deb`/`.msi`/`.exe` по каждой ОС.
- **Релиз — черновик** (`releaseDraft: true`): после зелёной сборки ассеты проверяются глазами
  и публикуются вручную.
- **FFmpeg в CI не ставится** — Tauri собирает только оболочку, `ffmpeg`/`ffprobe` вызываются
  у пользователя в рантайме (системные бинарники).
- **🔒 Без подписи / без секретов (осознанно):** подпись и нотаризация не настроены (open-source
  v1) — пользователь увидит предупреждение ОС «неизвестный разработчик». Это норма на старте;
  подпись — отдельная задача (платные сертификаты + секреты). Новых секретов нет: только
  встроенный `GITHUB_TOKEN` (правило #1 CLAUDE.md соблюдено).
- **Проверка — реальным прогоном Actions** (локально нельзя, нужны 3 ОС): сперва ручной
  `workflow_dispatch` с тестовым тегом → 4 job'а зелёные, ассеты в черновом релизе → затем
  боевой тег v1.0.0. Юнит-тестов нет (запись в `tests/README.md`).
- **Доки:** новая папка `.github/workflows/` с `README.md`; строка в карту `docs/README.md`.
  Версии не бампались (остаётся 1.0.0).

## [1.0.0] — 2026-06-22

🎉 **Первый публичный релиз FFNodes.** Полный цикл «файл → пайплайн из нод →
сгенерированная FFmpeg-команда → рендер с превью», мульти-аутпут (1 вход → N выходов),
сохранение/загрузка проектов (`.ffvproj`) и пресетов, приветственное окно при старте.
Итог роадмапа UX (Спринты 1–5) поверх рабочего рендера. Кроссплатформа Mac/Win/Linux
(Tauri 2). Ниже — что вошло в этот релиз.

### Спринт 5 — приветственное окно
- **Точка входа вместо сразу пустого холста.** При холодном старте показывается
  приветственный экран (`src/widgets/WelcomeScreen/`) с выбором: **Новый проект** /
  **Открыть проект** + список **последних проектов**. После любого действия — холст;
  назад к приветствию в рамках сессии не возвращаемся (UX-минимализм).
- **«Новый проект» = сразу чистый холст** (вход→выход, граф уже инициализирован в
  `useGraph`), без диалога сохранения; имя «проект без названия», сохранение позже через
  TopBar. **«Открыть»** — существующий системный диалог; **клик по недавнему** — открытие по
  пути (пропал/битый файл — мягко: убираем из «Недавних», остаёмся на приветствии).
- **Гейт показа — флаг `started` в `App`** (`false` → приветствие). Хуки общие с холстом
  (приветствию нужны те же `recent`/`openProject`), поэтому гейт ранним `return` внутри App,
  без размонтирования. Переход на холст при открытии — **только при успехе** (отмена диалога
  оставляет на приветствии).
- **Переиспользован фундамент Спринта 4 без дублирования:** список недавних
  (`useRecentProjects` + `recent.json`, тот же, что в выпадашке TopBar), `useProject`
  (open/new), `useGraph.loadGraph`, тип `RecentProject`. `WelcomeScreen` — чистая презентация
  (рендер из пропсов, действия наружу колбэками).
- 🔒 **Security:** новой поверхности нет. Открытие недавнего делегирует в существующий
  `useProject.openProjectFromPath` → `deserializeProject`, где защита путей из недоверенного
  файла (`safePath`, N-004) уже стоит со Спринта 4. Имена/пути недавних рендерятся как текст
  (React-автоэкранирование). security-review — «дыр нет».
- **Проверено вживую** (Vite + Playwright): приветствие при старте, «Новый проект» → чистый
  холст с «проект без названия». Тест компонента не пишем (в проекте нет Testing Library —
  все фронт-тесты на чистые функции); пробел зафиксирован в `tests/README.md`.

### Спринт 4 — сохранение/загрузка проекта + пресеты
- **Файл проекта `.ffvproj`.** Всё состояние холста (вход + все выходы + их настройки)
  сохраняется в файл и открывается обратно. Сценарий: 1 вход + N выходов со своими настройками
  → сохранил → в другой раз открыл и рендеришь. Формат — JSON с версионированием схемы
  (поле `version`: старые проекты прогоняются через миграцию, неизвестная будущая версия →
  понятная ошибка, не краш).
- **Сериализация (чистый модуль `shared/lib/project/`).** `serializeProject` выдёргивает из
  React Flow nodes/edges только сохраняемые поля (тип, позиция, deletable, params/filterId
  фильтра, путь доп. входа, targetHandle merge-нод); рантайм (колбэки, info, invalid) не пишем.
  `deserializeProject` — встречная, со строгой валидацией формы/версии (не доверяем содержимому
  файла: чужой/битый/будущий → `ProjectFormatError`; мусор внутри → предупреждение, не краш).
  Round-trip граф→JSON→граф покрыт тестами (мульти-аутпут + merge).
- **Tauri save/open.** Rust `write_project_file`/`read_project_file` (путь из системного
  диалога), `useProject` оркестрирует serialize→запись / чтение→deserialize→восстановление
  графа (`useGraph.loadGraph` доклеивает колбэки) и входа. Кнопки «Открыть»/«Сохранить» + имя
  проекта в TopBar.
- **Пути.** Хранятся абсолютными. Файл входа переехал/удалён при открытии — не краш: ffprobe
  не прочитает, показывается ошибка, путь остаётся для контекста, юзер перевыбирает.
- **Пресеты (настройки одной выходной ветки).** `extractBranch` собирает линейную ветку выхода
  (операции + params, без привязки к файлу; слияние/merge не сохраняется), `applyPreset`
  применяет к выбранному выходу. Хранятся отдельными `.json` в app-config/presets. Панель
  «Пресеты выхода» в каталоге: применить к выбранному выходу / сохранить ветку / удалить.
- **Список последних проектов (задел под приветственное окно).** `recent.json` в app-config
  (последние 10), выпадашка «Недавние» в TopBar — открыть проект из списка.
- **🔒 Security-review:** пути входов из НЕдоверенного `.ffvproj` пропускаются через `safePath`
  (N-004) в deserialize — путь-«флаг» (ведущий «-») не станет опцией ffmpeg при рендере; имена
  пресетов санируются в Rust (`sanitize_file_stem`) против path traversal; содержимое файлов не
  интерпретируется слепо.
- **Код-ревью (high) → фиксы:** инвариант «основной вход/выход неудаляемы» восстанавливается в
  `loadGraph` принудительно (не доверяя deletable из файла); `applyPreset` проверяет существование
  выбранного выхода (защита от async-гонки «выход удалён, пока пресет читался»).
- **Тесты:** фронт 254 ✅, Rust 19 ✅.

### Спринт 3 — мульти-аутпут (1 вход → N выходов)
- **Один вход → несколько выходов одной командой (Вариант A).** Сценарий: взял видео и
  выгнал сразу в нескольких форматах/разрешениях (напр. 1080p H.264 + 720p H.265 + GIF).
  Один декод входа, поток ветвится через `split`/`asplit` на N выходных секций
  (`-map … outputArgs out_i`) — эффективнее N параллельных запусков. У каждого выхода свои
  кодеки/настройки, заданные нодами его ветки.
- **Развилка A vs B** (одна команда с N -map vs N запусков) разобрана и утверждена в пользу
  A: лучше эффективность (декод один раз), естественно ложится на наш filter_complex со
  split-лейблами (как GIF-палитра), меньше трогает Rust (`RenderState` остаётся на один процесс).
- **Модель графа:** снято допущение «выход один». `dag.topoSort` обходит N выходов с проверкой
  целостности в обе стороны (каждый вход доходит до выхода, каждый выход достижим из входа);
  `outputNodes()`; `isLinearGraph` → false при >1 выходе. Кнопка «Выход» в каталоге добавляет
  выходную ноду (доп. выходы deletable, основной — нет); ветку в выход тянет пользователь.
- **Генератор:** `buildMultiOutputPlan` (список выходов + аллокатор split: поток, читаемый
  >1 раз, размножается — иначе filter_complex невалиден), `generateMultiOutputCommand`;
  `GeneratedCommand.outputPlaceholders` (`output_0.mp4`…).
- **Рендер/прогресс:** save-диалог на каждый выход (отмена любого отменяет рендер),
  `substituteOutputs` подставляет пути; одна линия прогресса (FFmpeg даёт общий `out_time_us`
  на команду). Rust `run_ffmpeg` принимает `output_paths` — при отмене чистит ВСЕ недописанные
  файлы. Зонд всех результатов → метаданные «После» по выходам.
- **Превью/predict по выбранному выходу:** `predictOutput(…, outputNodeId)` считает по ветке
  конкретного выхода; каждая выходная нода на холсте показывает СВОИ характеристики (раньше —
  одинаковые у всех). Вкладки выходов в панели «До/После» + кадр «После» выбранного выхода.
- **Код-ревью (high) → фикс N-018:** мульти-аутпут + overlay (merge без аудио) в одной ветке
  давал неподключённую asplit-ветку → ffmpeg падал. Исправлено (счёт потребления по
  потребителю). Заведён открытый техдолг: N-019 (validateGraph для filter_complex-путей),
  N-020 (af-фильтры в DAG), N-021 (дубль планировщиков) — вне скоупа спринта.
- **🔒 Security-review проведён** (тронули запуск FFmpeg + выходные пути): все N путей идут
  через `safePath` (N-004), ffmpeg запускается массивом без shell, чистка при отмене удаляет
  только выбранные пользователем выходы — **новых дыр нет**.
- Тесты: фронт 228 ✅ (+22: dag/generate/build split/предсказание по выходу/substituteOutputs),
  Rust 16 ✅. `npm run build` зелёный.

### Спринт 2 — дефолты в нодах из метаданных входа
- **Параметры новой ноды заполняются разумными значениями из входа**, а не статичной
  заглушкой. Хедлайн: вход H.264 → добавил «Сменить кодек» → в ноде уже стоит **H.264**,
  осталось переключить на H.265. Везде выбор кликом (выпадашки/числа), не свободный ввод.
- **Единый механизм** — необязательное поле `FilterDef.defaultsFromInfo(info)` (чистая
  функция «info → частичные параметры»), накладывается ПОВЕРХ статичных `default` в
  `useGraph.addFilterNode`. Без хардкода в нодах. Дефолт берётся в момент ДОБАВЛЕНИЯ ноды;
  смена файла позже уже добавленные ноды не пересчитывает (предсказуемо, не затирает правки).
- **Операции с дефолтом из входа** (прошёлся вручную по всем 29): «Сменить кодек» (текущий
  кодек), «Кадрировать» (область = полный кадр), «Отступы/леттербокс» (размеры входа),
  «Сменить частоту кадров» (fps входа), «Обрезать по времени» (конец = длительность).
  Остальные 24 — без дефолта осознанно (нет параметров либо творческий/качественный выбор).
- **Защита от мусора:** ключ возвращается только при ненулевом поле info (иначе статичный
  дефолт). `fps` отсекает 0 (VFR/битый контейнер дал бы невалидный `fps=0` и падение рендера).
- Пункт роадмапа «панель только выбранной ноды» снят: премиса не совпала с кодом (параметры
  рисуются ВНУТРИ нод на холсте, а не в левой панели) — уточнено в `docs/IDEAS.md`.
- Тесты: фронт 206 ✅ (+14: convert/trim/resize defaults). Security: поверхность не тронута
  (читаем доверенный MediaInfo, аргументы команды формируются как раньше) — дыр нет.

### Спринт 1 — дешёвые UX-победы (пункты 1–3)
- **Drag & drop файла.** Видеофайл можно перетащить прямо в окно (панель превью) — грузится так
  же, как через диалог (ffprobe + метаданные + кадр «До»). Tauri v2 `onDragDropEvent` отдаёт
  реальный путь файла; путь проходит `safePath` (N-004, как пути из диалогов) и фильтр по
  видеорасширению. Оверлей «Отпусти файл здесь» при перетаскивании; не-видео → понятная ошибка
  (уже выбранный файл при этом не сбрасывается). Пока один файл (батч — позже). Новые
  `useFileDrop`, `videoExtensions.ts` (единый список расширений для диалога и drop).
- **Навигация по холсту.** Панорама перенесена на ПРАВУЮ кнопку мыши (`panOnDrag={[2]}`) — ЛКМ
  освобождена под таскание нод и протяжку связей; ЛКМ по пустому холсту = рамка выделения
  (`selectionOnDrag`); панорама колесом/трекпадом (`panOnScroll`); нативное контекстное меню на
  ПКМ погашено. **Хэндлы соединения крупнее** (~7→14px + бордер под фон) — легче попасть в
  «крючок»; общий класс `shared/ui/handleClass.ts` во всех 4 типах нод.
- **Размеры окошек в каталоге функций.** Пункты каталога ужаты и выстроена иерархия:
  заголовок раздела — «оглавление» (полужирный, UPPERCASE, у левого края, главнее пунктов);
  вложенные пункты — легче (название 13px, описание 11px вплотную), с крупным отступом слева
  (`pl-6`) для чёткой «лесенки» вложенности. Раскрытый раздел: его пункты лежат на более
  тёмной панели (`bg-bg/40`) во всю ширину каталога (`-mx-2`) — список визуально утоплен под
  заголовком. Описание под названием сохранено (обучающий слой). Подобрано по скриншотам-эталону.
- Тесты: фронт 192 ✅ (+4 на отсев не-видео). Security-review (приём путей из drop): дыр нет.

### Оценка размера файла (этап «estimate-size», N-010)
- **Размер «После» теперь считается, а не копируется от входа** (раньше показывал тот же
  размер, что «До»). Оценка: `(видео_битрейт + аудио_битрейт) × длительность / 8`.
- **Операции корректируют битрейт:** scale/pad/fps — пропорционально пикселям×fps; «Сжать
  видео» — битрейт из CRF (~5 Мбит/с на 1080p30 при CRF23, шаг 6 CRF = вдвое); «Сменить
  кодек» — H.265 ×0.6 / VP9 ×0.65 (экономия при том же качестве); «Извлечь аудио»/«Убрать
  звук» — обнуляют соответствующую дорожку. trim/speed меняют длительность → размер
  пересчитывается.
- **Честность:** если операции не повлияли на битрейт/длительность — показываем РЕАЛЬНЫЙ
  размер входа (точнее оценки). Остаётся оценкой (помечена «≈») — зависит от контента; после
  рендера панель показывает точный размер (ffprobe).
- Чистый `size.ts` (`estimateSize`/`scaleVideoBitrate`/`estimateBitrateFromCrf`). Тесты:
  фронт 188 ✅ (+17). Проверено реальным ffmpeg (порядок битрейта).

### Multi-input графы: DAG + filter_complex (этап «multi-input», N-008, N-016)
- **Переход с линейной цепочки на полноценный DAG:** несколько входов, ветвление и слияние
  потоков. Граница «простой `-vf` путь vs `filter_complex`» — единственный рубильник
  `dag.isLinearGraph(graph)`: filter_complex включается, только если есть merge-операция,
  больше одного входа или ветвление/слияние рёбер. Линейные графы и все 27 фильтров работают
  байт-в-байт как раньше.
- **Качественный GIF (N-008 закрыт):** «Сделать GIF» перешёл на двухпроходную палитру
  `split→palettegen→paletteuse` через filter_complex — заметно чище цвета. Это single-input
  merge-операция (второй `-i` не нужен).
- **Новые операции «Слияние»:** «Наложить поверх» (overlay — водяной знак/лого/картинка,
  второй вход) и «Склеить ролики» (concat — два ролика подряд, сумма длительностей).
  Merge-нода рисует ДВА именованных входа («Основное видео»/«Накладка», «Первый/Второй ролик»).
- **Добавление входа:** кнопка «Добавить вход» в каталоге; дополнительный вход выбирает файл
  на самой ноде. В один входной хэндл ведёт одно ребро (новое подключение заменяет старое).
- **Предсказание «До→После» при слиянии:** топо-обход с накоплением характеристик; overlay
  сохраняет размер основного видео, concat суммирует длительность.
- **Превью-кадр «После» для DAG (N-016 закрыт):** строится через filter_complex (Rust
  `extract_frame_complex` рядом со старым `extract_frame`). GIF/overlay/concat показывают
  реальный результат.

### Рендер + линия прогресса (этап «render») — полный цикл «файл → результат»

- ⚠️ Полный клик (Рендер → save-диалог → файл) проверяется вручную (нативный диалог);

### Генератор команды + параметры нод (этап «command-generator»)
- **Сердце проекта:** `src/shared/lib/ffmpeg/generate.ts` — чистая функция `Graph → {args, display}`.
  Обходит граф от input по связям до output (MVP — линейная цепочка, опция `-vf`), собирает фильтры
  в порядке связей. Команда = массив аргументов (защита от инъекций). Неполный/циклический граф →
  понятная ошибка, без падения. Тесты — Vitest table-driven (8: цепочки, разрыв, цикл, без фильтров).
- Ноды `InputNode` (синяя) / `OutputNode` (зелёная); `FilterNode` получил редактируемые поля параметров.
- `useGraph`: стартовая пара вход→выход (неудаляемые); добавленный фильтр встраивается в цепочку
  перед выходом с авто-перецепкой связей; `onParamChange` пишет значения в `data.params`; маппинг
  состояния React Flow → доменный `Graph`; генерация команды через `useMemo`.
- `CommandBar` показывает реальную сгенерированную команду (или подсказку при неполном графе),
  копирование в буфер; обновляется в реальном времени.
- Проверено вживую (Playwright): клик→фильтр в цепочке, правка параметра → команда меняется
  мгновенно (`scale=640:-2,fps=30`), консоль чистая. N-003 (удаление среднего фильтра рвёт цепочку) — в CODE_NOTES.
